# Codex 量産ガイド — 外部ツールでのゲーム生成ワークフロー

Codex CLI（や他の外部LLMツール）で Swizzle ゲームJSONを量産するためのガイドです。
フォーマットの詳細は [SWIZZLE_JSON_SPEC.md](../../SWIZZLE_JSON_SPEC.md) が唯一の正であり、
本書はワークフローと品質基準に絞って説明します。

> 関連: ChatGPT向けには [CHATGPT_MASTER_PROMPT.md](./CHATGPT_MASTER_PROMPT.md) /
> [CHATGPT_NETA_PROMPT.md](./CHATGPT_NETA_PROMPT.md) があります。

---

## ワークフロー

1. **SPECを読み込ませる**
   `SWIZZLE_JSON_SPEC.md` 全体をコンテキストに入れる。特に以下のセクションは生成前に必ず参照させること:
   - アセット様式（アートディレクション）（画像・SVGの様式定義）
   - 品質チェックリスト（提出前に必ず確認）
   - 禁止パターン集（自動検証で落ちる）
   - メカニクス・ガイド

2. **ネタを1件選ぶ**
   `src/ai/v2/neta.json`（または `neta_*.json`）から未消化のエントリを選ぶ。
   ネタの `mechanic` フィールドを尊重し、勝手に別メカニクスへ置き換えない。

3. **4つの基本パターン内で設計する**
   コアメカニクスは「タップ判定 / ドラッグ配置 / スワイプ方向 / 長押しチャージ」
   （またはその組み合わせ）に収める。物理演算・NPC AI前提の設計は実装不可。

4. **難易度を決めてパラメータに反映する**
   `src/ai/v2/mechanics-catalog.json`（v1.1）の `difficulty` / `skillTypes` を参照。
   - easy: 大きいターゲット / speed 2〜4 / duration 7〜10秒 / 1アクション成功
   - normal: 標準 / speed 4〜7 / duration 6〜8秒
   - hard: 狭い判定窓（0.3秒以上）/ speed 7〜10 / duration 5〜7秒

5. **アセット様式（アートディレクション）に従う**
   画像・SVGアセットを作る場合は、SPECの「アセット様式（アートディレクション）」節に**必ず**従う。
   様式名「GALLERY」= 無彩色基調 + アクセント1色（既定ヴァーミリオン `#FF3B1F`）、影なし・
   フラットベクター・余白多め・装飾過多にしない。色はパレットのHEXを直接使い、
   オブジェクトは白背景・単一・中央、背景は中央を空ける。
   （この様式は `src/ai/v2/art-direction.json` が定義源で、生成側・画像QAも同じ基準を参照する）

6. **チェックリストで自己検証する**
   SPECの「品質チェックリスト」を1項目ずつ確認させてから出力する。
   特に頻出の失敗（下記）は出力前に潰すこと。

7. **1ゲーム = 1 JSONファイルで出力する**
   `project` ルートを持つ完全なJSONのみを出力（コードブロックや解説文を混ぜない）。

## レビューで落ちるポイント（人間審査の観点）

アップロードされたゲームは管理者がエディターの **AIゲームレビュー**（GameReviewQueue）で
実際にプレイして審査します。審査時の却下理由タグは以下です:

| タグ | 意味 | よくある原因 |
|------|------|------------|
| 動かない | ゲームが起動しない/即終了 | JSON構造エラー、必須フィールド欠落 |
| タッチ反応しない | 操作が効かない | touch条件のtargetId間違い、touchableでないオブジェクト |
| 仕様違い | ネタ/説明と挙動が違う | メカニクスの勝手な置き換え |
| バランス悪い | 難しすぎ/簡単すぎ | タップ数過多（秒数×3超）、判定窓が狭すぎ（0.3秒未満） |
| ゲームにならない | 目標・操作・判定が不明確 | 4つの明確性の欠如、自動成功 |

画像はレビューUIのアセットプレビュー（背景+全オブジェクトのサムネイル）でも確認されます。
オブジェクト画像は**白背景・単一オブジェクト・フラットスタイル**に加え、
SPECの「アセット様式（アートディレクション）」（無彩色基調 + アクセント1色・影なし）を守ること。

## 自動検証で頻出する失敗コード TOP

1. `AUTO_SUCCESS` — time→success のみで成功する（touch/collision を成功パスに入れる）
2. `COUNTER_UNREACHABLE` — 「同じものをN回タップ」設計（N個の別オブジェクトに変える）
3. `OBJECT_NO_RULES` — ルールのないオブジェクト（全オブジェクトにルールを書く）
4. `SUCCESS_FAILURE_CONFLICT` — 同一条件が成功と失敗を両方発火
5. `MISSING_SOUND_ID` — se_tap / se_success / se_failure の定義漏れ

完全な一覧と対処は SPEC の「禁止パターン集」を参照。

## アップロード後の品質ゲート

- AI量産パイプライン経由のゲームは総合スコア **70点未満で自動的に審査キュー行き**
  （`is_published=false`, `review_status='pending_review'`）になります。
- 手動アップロード（ReviewQueue.tsx のローカルJSONフロー）では、レビュー合格時に
  公開されます（`review_status` はデフォルトで `approved`）。
- 審査待ち件数は `npm run ai:status` で確認できます。
