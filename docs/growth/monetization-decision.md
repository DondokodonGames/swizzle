## 0. 前提整理 — 4手段は「誰に課金するか」が違う

実装済みの4手段は、しばしば混同されるが**課金対象とUX上の位置が異なる**。
ここを揃えないと意思決定を誤る。

| 手段 | 課金対象 | UX上の位置 | 実装の事実 |
|---|---|---|---|
| **Pay-Per-Play ウォレット** | **クリエイター**（作成行為） | エディタ（作成時） | `on_game_created_consume_credit` トリガーが `user_games` INSERT 時に消費。100作成まで無料→以降 ¥1/作成。チャージ ¥100/500/1,000/3,000（`TOP_UP_OPTIONS`） |
| **Free / Premium サブスク** | プレイヤー兼クリエイター（ヘビー層） | 全体 | $4.99/月。Premium = 広告非表示・無制限作成・全テンプレ・上級編集・アナリティクス・HTML5書き出し |
| **AdSense 広告** | **プレイヤー**（無料回遊） | ブリッジ画面（ゲーム間） | `AdUnit` が Free ユーザーにのみ表示。Premium は非表示。スロット未設定時はプレースホルダー |
| **NFC/QR + Payment Links** | **プレイヤー**（特定ゲーム） | 物理ロケーション経由の単体プレイページ | `game_payment_config` で価格設定、`one_time_access` トークンで回数管理。`PlayGamePage` がゲート表示 |

**核心の含意**: 「ゲーム版TikTok」の中核体験 = **縦スワイプ連続プレイ（回遊ループ）**。
このループに直接課金ゲートを挟むのは Pay-Per-Play でも NFC でもなく、本来どちらも回遊ループの外にある。
回遊ループそのものは **広告**（無料維持）と **Premium**（広告除去アップセル）でしか触れていない。

## 1. ゲーム版TikTok路線での適合性比較

評価軸:
- **回遊との相性**: 縦スワイプ連続プレイのテンポを壊さないか
- **摩擦**: ユーザーが財布を意識する回数・タイミング
- **単価/原資**: 1アクションあたりの期待収益と、その持続性

| 手段 | 回遊との相性 | 摩擦 | 単価/原資 | 総評 |
|---|---|---|---|---|
| **広告（AdSense）** | ◎ ブリッジ画面は元々“間”の時間。広告挿入の自然な枠 | 低（受動的） | 低単価×高頻度。日本 RPM 低め（$1–3/1k imp 想定）だが回遊数に比例 | **回遊最大化の主軸候補**。連続プレイ数がそのまま在庫になる |
| **Premium サブスク** | ○ 広告除去＝回遊体験の向上として売れる。作成上限解放はクリエイター訴求 | 低（任意） | 高単価×低頻度（$4.99/月）。LTV 高いが転換率は数% | **ヘビー層の主軸**。回遊を阻害せずアップセル |
| **Pay-Per-Play（作成）** | △ プレイヤー回遊には無関係。クリエイターの作成コスト | 中（作成100件後に発生） | ¥1/作成。原資というより**乱造抑制＋サーバ原価回収** | **回遊主軸にしない**。作成スパム防止＋AI原価の薄いカバーに用途を限定 |
| **NFC/QR ロケーション** | △ オンライン回遊とは別チャネル（店舗・イベント） | 中（明示的購入） | 中〜高単価×低頻度。特定ゲーム単位 | **物理チャネル専用**に縛る。オンライン回遊に持ち込まない |

## 2. 推奨シナリオ（仮説）

### 推奨: 「無料＋広告で回遊を最大化 / Premiumでヘビー層 / Pay-Per-PlayとNFCは回遊ループ外に隔離」

```
┌─────────────────────────────────────────────────────────┐
│ プレイヤー回遊ループ（縦スワイプ連続プレイ）            │
│   → 完全無料。ブリッジ画面に AdSense（主軸収益）        │
│   → 広告がうざいヘビー層に Premium($4.99/月)でアップセル │
├─────────────────────────────────────────────────────────┤
│ クリエイター（作成）                                    │
│   → 100作成まで無料。以降 Pay-Per-Play ¥1/作成          │
│     （収益目的ではなく乱造抑制＋AI原価の薄いカバー）    │
│   → 本格クリエイターは Premium で無制限作成             │
├─────────────────────────────────────────────────────────┤
│ 物理ロケーション（NFC/QR）                              │
│   → 特定ゲームの有料プレイ。オンライン回遊とは別KPIで管理 │
└─────────────────────────────────────────────────────────┘
```

**理由（3点）**:
1. TikTok型プロダクトの生命線は **retention × セッション長**。中核回遊に課金摩擦を一切入れず、
   無料×広告で連続プレイ数（= 広告在庫）を最大化するのが王道。
2. Premium は「広告除去」という**回遊体験の改善**として売るので、回遊を阻害せずに収益化できる
   （PtP のように体験を止めない）。
3. Pay-Per-Play と NFC を回遊ループから隔離することで、「いつ財布を意識させるか」を
   クリエイト時・物理接触時の**能動的な瞬間だけ**に限定できる。

### この推奨に Yes/No で答える論点（意思決定者向け）
- [ ] **広告を回遊の主軸収益と認める**か？（No の場合 = Premium 中心の薄利モデルになる前提を確認）
- [ ] **Pay-Per-Play を収益源と見なさず「乱造抑制＋原価カバー」に役割を限定する**ことに合意するか？
- [ ] **NFC をオンライン回遊KPIから切り離し、別チャネルとして別管理**することに合意するか？

## 4. クリエイター収益分配 — 設計案のみ（実装しない）

> **実装はしない。** 着手条件（後述）を満たすまでは設計スケッチのまま凍結する。

### 4.1 分配原資（どこから払うか）
- **広告収益プール方式**: 当月の AdSense 純収益（プレイヤー回遊由来）の一定割合 `R`（例: 30–50%）を
  クリエイター分配原資とする。Premium 収益は当面プラットフォーム側に留保（広告除去の対価のため）。
- 為替・AdSense 入金遅延を考慮し、**確定は月締め＋翌々月払い**を想定。

### 4.2 按分方法（誰にいくら）
候補2案。初期は **A案（プレイ数加重）** を推奨（最も改ざん耐性が高く説明可能）。

- **A案: 完走プレイ数加重**
  `share_i = pool × (completed_plays_i / Σ completed_plays)`
  完走 = `play_end` で `result <> 'skip'`（WP41 イベント基準）。スキップや自己プレイ除外で水増し対策。
- **B案: エンゲージ加重（プレイ数＋いいね）**
  `weight_i = completed_plays_i + α × likes_i`（α は調整係数, 例 5）。
  いいねは安価に水増しできるため α は小さく。初期はA案、データが出てからBへ移行検討。

### 4.3 最低支払額・不正対策
- **最低支払額**: ¥3,000 未満は翌月繰り越し（送金手数料・税務処理コストの下限）。
- **自己プレイ除外**: `analytics_events.user_id = user_games.creator_id` のプレイは分配計算から除外。
- **異常検知**: 1セッション/1ユーザーからの同一ゲーム反復は完走1回までに丸める。

### 4.4 必要なテーブル設計スケッチ（実装時の叩き台）
```sql
-- 月次の分配計算結果（確定スナップショット）
create table creator_payouts (
  id            uuid primary key default gen_random_uuid(),
  creator_id    uuid not null references auth.users(id),
  period_month  date not null,              -- 月初日（例 2026-07-01）
  completed_plays integer not null default 0,
  weighted_score numeric not null default 0,
  gross_share_yen integer not null default 0,
  status        text not null default 'pending'
                  check (status in ('pending','carried_over','paid','failed')),
  created_at    timestamptz not null default now(),
  unique (creator_id, period_month)
);

-- 分配の入力となる原資・パラメータ（監査用に固定）
create table payout_periods (
  period_month   date primary key,
  ad_net_revenue_yen integer not null,
  pool_ratio     numeric not null,          -- R（0.3〜0.5）
  min_payout_yen integer not null default 3000,
  finalized_at   timestamptz
);
```
- データソースは **WP41 の `analytics_events`**（`play_end` 完走集計）。新規イベント追加は不要。
- 送金実行は Stripe Connect（Express アカウント）想定だが、**本設計の範囲外**。
