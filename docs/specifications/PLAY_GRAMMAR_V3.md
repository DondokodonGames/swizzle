# PLAY GRAMMAR v3 — テキストで説明しない「遊びの文法」

対象: `src/ai/code/examples/*.js` の全ゲームと今後の新規ゲーム。
本書は **v3 リファクタ(WP61〜63・WP56改訂)の正典**。メイドインワリオ(5秒で伝わる1動詞)、
リズム天国(音が先生)、マリオパーティ(開始と結末の祭り)から「遊びの伝え方」を移植する。

> v2(GAME_QUALITY_STANDARD_V2)との関係: v2 は「手応え・見た目・音の最低保証」。
> v3 はその上に「**説明文を読まなくても、見れば遊べる**」を積む。矛盾したら本書が優先。

関連: [GAME_QUALITY_STANDARD_V2.md](./GAME_QUALITY_STANDARD_V2.md) / [SANDBOX_API_V2.md](./SANDBOX_API_V2.md) / [MECHANICS_CATALOG_V2.md](./MECHANICS_CATALOG_V2.md) / [ARCADE_ART_DIRECTION.md](./ARCADE_ART_DIRECTION.md) / 台帳: `docs/work-plans/ledger/game-assignments.csv`

## 1. 5秒の文法(MUST)

1ゲーム=**1動詞**。プレイヤーは以下の5点を、**文章を1行も読まずに**理解できなければならない:

| # | 問い | 伝える手段 |
|---|------|-----------|
| (a) | 何が目的か | 最初のフレームに「目的の名詞」が見えている(狙う物・守る物・ゴール) |
| (b) | どう操作すれば何が起きるか | ATTRACTのゴースト実演 + 開始直後のお手本(§2.1, §2.2) |
| (c) | 何を達成すべきか | ゴールの可視化: 残りN個のアイコン列、ゴール旗、満タンになるゲージ |
| (d) | どうミスすると失敗か | 危険の記号性(§2.3)+ telegraph(§2.4) |
| (e) | なぜ失敗したか(納得) | 失敗の因果提示: 当たった物を止めて見せる(§2.5) |

**チェック方法**: ATTRACT画面のスクリーンショット2枚(コンタクトシート)を初見の人間に見せ、
「何をするゲームか」を言い当てられるか。言い当てられなければ (a)(b) が不合格。

## 2. テキストレス伝達 技法カタログ

書き換えセッションは、担当ゲームのメカニクスに合わせて以下から**必ず (2.1 + 2.3 + 2.4 + 2.5) を実装**し、
該当があれば 2.2 / 2.6 / 2.7 を加える。

### 2.1 ATTRACTゴースト実演(MUST)

ATTRACT中、**AIが実際にゲームロジックを1サイクル遊んで見せる**。静止画+説明文のATTRACTは禁止。

- 手カーソル(WP62 で `game.draw.hand(x, y, {press})` としてエンジン提供。実装前は 8px ドットの手スプライトを自作)が
  画面内を動き、タップ位置でリング演出 → ゲーム内の実オブジェクトが実際に反応する
- 成功例を1回 + (危険があるゲームは)失敗例を1回、計3秒以内のループ
- デモはゲームの update/描画コードを流用する(見た目が本番と同じであること)。別描画の「デモ風アニメ」は禁止
- HOW_TO_PLAY 等の説明文変数・描画は**全廃**する

### 2.2 開始時お手本(推奨 — 初見殺し防止)

PLAYING 冒頭 0.5〜1.0 秒、最初のターゲットを強調パルス(白縁の明滅+拡縮)し、
タップすべき位置に手カーソルを1回だけ出す。ゲームはその間も動いてよい(停止演出は不要)。

### 2.3 記号文法(MUST — ARCADE_ART_DIRECTION §記号文法と同一)

| 意味 | 記号 |
|------|------|
| 狙う物・押す物 | **明滅 + 白縁**、待機バウンス(押せる物は揺れる) |
| 危険物 | **赤系 + トゲ形状**。触れる前に必ず予告(§2.4) |
| ゴール | 旗・ゲート・光る出口(画面内に常時見える) |
| 残り時間 | 収縮するリング/バー(数字だけにしない) |
| 進捗 | 残りN個のアイコン列(消し込み)、伸びるゲージ、高度メーター |
| 操作ゾーン | 押せる領域は明度を上げ、縁を光らせる |

### 2.4 telegraph — 危険の事前予告(MUST)

**予告なしの即死は禁止**。危険は発生の 0.5〜0.8 秒前に必ず予告する:
出現位置の点滅マーカー / 落下物の影 / 警告線(レーザーの照射前線) / 画面端の「!」アイコン + `se_bad` 系の予告音。

### 2.5 失敗の因果提示(MUST)

被弾・失敗の瞬間に **hit-stop 0.3〜0.6秒**: ゲームを止め、当たった物・触れた物を白フラッシュ+拡大でハイライトし、
`game.feedback.bad` を鳴らしてから RESULT に遷移する。「何に当たって死んだのか」を見せずに RESULT に飛ぶことを禁止。
成功側も同様に、達成の瞬間の対象物をハイライトしてから CELEBRATION に入る。

### 2.6 リズム天国原則 — 音が先生(該当ゲームは MUST)

- `game.audio.melody` の拍にゲーム内イベント(出現・移動・判定窓)を同期させる。**音を聞くだけで次のアクションのタイミングが予測できる**
- 判定は段階制: PERFECT / GOOD / MISS(ズレ ms で分岐)。判定語はホワイトリスト語(§3)
- 入力音は結果で音程を変える(成功=上昇音、失敗=濁音)。連続成功で音程が上がる(コンボの可聴化)

### 2.7 マリオパーティ原則 — 開始と結末の祭り(MUST)

- **READY? → GO!**: PLAYING 遷移直後に 0.8 秒のカウントイン(表示はホワイトリスト語+アートの点滅。3-2-1 は数字表示可)。GO! と同時に操作受付を明示
- RESULT の演出差: CLEAR = キャラが喜ぶ+光の祝祭(音と光を飽和)、GAME OVER = キャラが倒れる/悔しがる(§ARCADE の CELEBRATION / ニアミス規定と同一)
- ニアミスは数字で悔しがらせる: 「あと1個!」「98%」(fx.popup、ホワイトリスト§3の範囲)

### 2.8 同一メカニクス内の変化軸(MUST — クローン量産の禁止)

台帳のメカニクスは**管理用の分類ラベル**であって「この形に均せ」という意味ではない。
同じメカニクスのゲーム(例: timing_one_shot は123本ある)を教科書形のワンパターンに
均すことを禁止する。各ゲームは以下の軸で必ず差別化する:

1. **固有フックの保全・増幅(最重要)**: 台帳の hook 列(=ヘッダー2行目のそのゲーム固有の
   面白さ。「壁反射で予測しづらい」「偽物は見逃す」「加速していく針」等)を消さずに、
   むしろ演出の主役にする。フックを消して「ただのタイミングゲー」にしたら失格
2. **プレッシャー型**(台帳の variation 列): 加速型 / 物量型 / 精度型 / フェイント型 / 変拍子型 —
   難易度カーブの「上がり方」を型で差別化する
3. **スパイス**(台帳の spice 列): コンボ倍率 / フィーバータイム / 黄金ターゲット /
   サドンデス演出 / 二重課題 / 逆転ボーナス — 得点と緊張の「山場」を差別化する
4. **尺**(duration_target): 同メカニクスでも帯域内で意図的に散らしてある。指定値に従う
5. **テーマ・スタイルパック・BGM・判定モチーフ**: §ARCADE の記号文法は共通のまま、
   「何が動き、何を待ち、何が壊れるか」の絵をテーマで完全に変える

- variation / spice は第一候補の指定。そのゲームの遊びと本質的に矛盾する場合のみ、
  同じリスト内の別の型を選んでよい(assignments の notes に理由を記録)
- **バッチ内でのコード使い回し禁止**: 同一バッチの他ゲームと ATTRACT デモ・演出・
  難易度カーブのコードを共有(コピペ)しない。エンジンAPI呼び出しの定石(feedback 等)は除く

## 3. ゲーム内テキストのホワイトリスト(MUST)

**指導文・説明文は全廃**。`draw.text` で表示してよいのは以下のみ:

- 数字・カウンタ・スコア(`3/12`, `00920`, `98%`, `x8`)
- `SCORE` `BEST` `HI-SCORE`(値は `game.best` の実値のみ。ハードコード禁止)
- 判定語: `PERFECT` `GOOD` `MISS` `NICE` `NEW RECORD`
- 進行語: `READY?` `GO!` `CLEAR` `GAME OVER` `TIME UP` `FINISH`
- 筐体ブランド定型: `INSERT COIN` `► 100円 投入 ◄` `TAP TO START` `TAP TO CONTINUE` + ゲームタイトル(ピクセルロゴ)
- ニアミス定型: `あと1個!` 等の「あと+数値+単位」1語

**禁止例**: `TAP THE STAR` / `SWIPE TO SORT` / `タップで反転` などの操作説明、遊び方の文章全般。
`HOW_TO_PLAY` 変数は削除する。ルールはアートと実演(§2)だけで伝える。

## 4. 尺の帯域表(正典)

メカニクス族(MECHANICS_CATALOG_V2 の A〜G)ごとの適正 `MAX_TIME`。帯域内なら現状維持、
帯域外は台帳(`game-assignments.csv` の duration_target)の指定値へ変更する:

| 族 | 帯域 | 根拠 |
|----|------|------|
| A タイミング | 8〜15s | 1動詞の反復。ワリオ式は短いほど締まる |
| B 照準・発射 | 10〜20s | 照準→結果確認のサイクル×3〜5回 |
| C 回避・追従 | 15〜25s | 難易度カーブ(後半加速)に尺が要る |
| D ジェスチャー | 8〜15s | 1ジェスチャーの快感を薄めない |
| E 配置・整理 | 15〜25s | 考える時間を含む |
| F 観察・判断 | 8〜15s(memory_sequence / pair_match は 10〜20s) | 瞬間判断は短く、記憶は往復分 |
| G 対戦・パーティ | 15〜30s | 逆転の余地を残す |

※ この表を変更する場合は `scripts/build-game-ledger.ts` の `DURATION_BAND` と必ず同期させること。

## 5. 難易度の再調整(MUST)

前回の「1/10難易度」ナーフで **NEEDED=1(1アクションで即クリア)が79本**ある。遊びが消えているので戻す:

- **NEEDED=1 の原則禁止**。例外は「1つの行為そのものが遊び」の族のみ
  (timing_one_shot / hold_charge / hold_duration / freeze / reaction_duel / slingshot / flick_launch / trajectory / drop_timing / counting / size_judge)
- 目安: **尺10秒あたり有意アクション3〜8回**。失敗が現実に起こり得る密度にする
- 「初見でも成功に手が届く」哲学は維持する。上げるのは**行動回数**であって理不尽さではない
  (telegraph §2.4 と開始時お手本 §2.2 が難易度上昇の安全網)
- 台帳の `needed_action` 列の指示に従う。`維持` 以外は必ず再調整し、`// 修正` 注釈は削除してよい

## 6. 音の設計(MUST)

### 6.1 イベント→SE語彙マッピング

「タップ=se_tap、成功=se_success、失敗=se_failure」の3音固定を禁止し、イベントの意味で使い分ける:

| イベント | SE |
|----------|-----|
| 入力受理(中立) | `se_tap` |
| 良い結果(命中・正解) | `game.feedback.good`(=se_good+burst+popup) |
| 悪い結果(被弾・誤答) | `game.feedback.bad`(=se_bad+flash+shake) |
| 収集(コイン・アイテム) | `se_coin` |
| 跳躍・反転・発射 | `se_jump` |
| 中間目標・コンボ節目 | `se_milestone` |
| チャージ完了・強化 | `se_powerup` |
| 破壊・崩壊 | `se_break` |
| 決着ジングル | `se_success` / `se_failure`(結末専用。道中で使わない) |

**distinct SE 3種以上が必須**(スコアラー検査対象)。存在しない `se_*` IDはバリデーターがエラーにする(WP63)。

### 6.2 BGM

- 第一候補: `game.audio.melody` による**ゲーム固有メロディ**(テーマの雰囲気を音階・テンポで表現)
- 代替: テーマに合うプリセット `bgm_tense` / `bgm_cute` / `bgm_dark`。**`bgm_main` の惰性使用は禁止**(台帳の bgm_direction 列に従う)
- 緊張の可聴化: 残り時間わずか/ラストチャンスで警告音ループまたは melody のテンポ切替(ARCADE §3)

## 7. 筐体ブランド規定 — 維持と差別化の線引き

**維持する(全ゲーム共通・変更禁止)**:
- IIFE 構造、状態機械 ATTRACT → PLAYING → RESULT
- コイン投入演出(`► 100円 投入 ◄` / `TAP TO START` の点滅)、スキャンライン(8px)
- `game.end.success/failure` による終了、ファイル名・slug・番号

**差別化する(ゲームごとに固有・共通コピペ禁止)**:
- テーマ・世界観・キャラ(sprite)・パレット(スタイルパック)
- BGM(melody 固有化)・SE語彙・デモ実演の内容
- ATTRACT のピクセルロゴ、RESULT の祝祭/悔しがり演出
