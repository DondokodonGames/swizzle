**担当**: Opus（ドラフト作成）→ 人間（意思決定と実行） / **依存**: WP41/42と並行可 / **想定規模**: S（ドキュメント作成のみ）

以下をそのまま新しいOpusセッションに貼って実行する。**コード変更なし、`docs/growth/` へのドキュメント作成のみ。**

## 背景

Swizzle は「ゲーム版TikTok」を目指すミニゲームプラットフォーム。マネタイズ基盤は実装済み
（ウォレットPay-Per-Play / Free+Premiumサブスク / NFC物理ロケーション / AdSense）だが、
**どれを主軸にするかの意思決定**と、**マーケティング・コミュニティの実行計画**が存在しない。

既存の参考資料: docs/marketing/current_status.md（SNS自動化の構想、Twitter API $100/月等のコスト見積）、
src/marketing/（TikTok/Discord/Twitter自動投稿のstub実装 — API未統合）。

## やること

`docs/growth/` に3つのドキュメントを作成:

### 1. monetization-decision.md（主軸マネタイズの意思決定資料）
- 既存4手段（Pay-Per-Play / サブスク / 広告 / NFCロケーション）それぞれの「ゲーム版TikTok路線での適合性」を、
  ユーザー行動（縦スワイプ連続プレイ）との相性・摩擦・単価の観点で比較
- 推奨シナリオを1つ提示（例: 無料+広告で回遊を最大化し、ヘビーユーザーにPremium。Pay-Per-Playは
  NFC物理ロケーション専用に縛る — 等。データ無しの現段階での仮説と明記し、WP41のファネルデータで
  検証する判断基準を併記）
- クリエイター分配の**設計案のみ**（実装しない）: 分配原資・按分方法（プレイ数/いいね加重）・
  最低支払額・必要なテーブル設計のスケッチ。実装判断はクリエイター数が閾値を超えてから、の条件付き

### 2. community-playbook.md（コミュニティ立ち上げ手順 — 人間が実行）
- Discordサーバー構成案（チャンネル設計: 作品共有/フィードバック/ネタ募集/告知）と初週の運営タスク
- 「クリエイターファースト」の初期施策案: 審査通過ゲームの紹介、週間ベストゲーム、ネタコンテスト
- 既存の GameReviewQueue 審査フローとコミュニティの接続（審査通過→Discord告知の手動運用）

### 3. sns-playbook.md（SNS手動運用 — 自動化は将来）
- プラットフォーム優先順位（ショート動画との相性でTikTok/YouTube Shorts優先等）と投稿フォーマット案
  （ゲームプレイ録画10-15秒+結果画面）
- 週次の投稿カレンダーテンプレート
- src/marketing/ の自動化stubは**使わない**ことを明記（API統合は手動運用で効果確認後の将来WP。
  current_status.md のコスト見積 $150-220/月 を参照し、着手条件を書く）

## やらないこと

- コード変更・API統合・クリエイター分配の実装
- 広告出稿等の有料施策の実行（計画への言及のみ）

## 受け入れ基準

- 3ドキュメントがコミットされ、各々に「人間が次にやる具体的アクション」のチェックリストがある
- monetization-decision.md の推奨シナリオに、WP41のデータでの検証基準（数値）が付いている
- git diff が docs/ のみ

1. 人間が読んで意思決定できる粒度か（推奨にYes/Noで答えられるか）を確認
2. 完了時: 成果サマリ5行を出力
