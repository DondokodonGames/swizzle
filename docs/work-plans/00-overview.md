# Swizzle 改善ワークプラン — マスタープラン

4つの課題（量産の均質化 / アートディレクション / 技術負債 / マネタイズ・マーケ）を、
**1ワークパッケージ(WP) = 1セッション** で Opus / Codex / 人間に分割して順番に実行するための指示書群。

各WPファイルは **そのまま新しいセッションに貼って作業を開始できる自己完結プロンプト** になっている。

## 担当の振り分け原則

| 担当 | 対象 |
|------|------|
| **Opus** (Claude Code) | コード変更・設計・レビュー・ドキュメント作成のすべて |
| **Codex** | ゲームJSONの量産のみ（SWIZZLE_JSON_SPEC.md と CODEX_GENERATION_GUIDE.md を読んで生成） |
| **人間** | 案の選定、レビュー承認、外部アカウント設定（AdSense/Stripe）、マーケ実行 |

## 実行順序と依存関係

```
Track 1（品質）     11 ──→ 12        11 ──→ 13(Codex)
Track 2（見た目）   21 ──→ [人間: 案選定] ──→ 22
                    21選定 ──→ 23 ──→ 24(Codex)
Track 3（負債）     31 ──→ 32 ──→ [人間: 承認] ──→ 33
Track 4（収益）     41 / 42 / 43（相互依存なし、Track 3完了後を推奨）
```

- Track間は原則直列（品質→見た目→負債→収益）だが、**13/24のCodexバッチは他Trackと並行可**
- 22（UI刷新）と 33（リファクタ）は同じファイルに触れる可能性があるため**並行禁止**（22が先）

## セッション運用ルール（トークン節約）

1. **1WP = 1セッション**。WPファイルの内容を冒頭にそのまま貼る（探索を最小化するため、必要なファイルパス・行番号・背景は各WPに埋め込み済み）
2. ブランチ: `claude/wp-<番号>-<短い名前>`（例: `claude/wp-11-prompt-diet`）。完了したらPR or mainへのマージは人間が判断
3. WP完了時、担当エージェントは **成果サマリ5行 + 変更ファイル一覧** を出力する。次のWPセッションにはそのサマリだけを渡す（前セッションの全文脈は持ち込まない）
4. 完了したらこのファイルのチェックボックスを埋めるコミットを入れる
5. **全WP共通の受け入れ基準**: `npx tsc --noEmit --skipLibCheck` クリーン / `npm run test` 全件green（現在354件）/ `src/types/editor/contract.ts` と `src/services/rule-engine/` は WP32の承認なしに変更しない

## 品質の守り（既存資産 — 各WPはこれを壊さないこと）

前タスクまでに整備済みの安全網。プロンプトや見た目を変えても品質はこれらが守る:
- **類似度ゲート**: 既存ゲームと70%超類似のコンセプトを自動リジェクト（Orchestrator + GamePatternAnalyzer）
- **フェーズ層**: 多段階ゲームを状態機械で宣言→PhaseCompilerがフラグ+ルールへコンパイル、グラフ検証でsuccess到達性を保証
- **DryRunSimulator**: プレイ可能性の事前検証（アセット生成前ゲート）
- **画像QA**: Claude visionによる生成画像の採点と再生成（ImageQualityChecker）
- **公開ゲート**: 総合スコア70未満は pending_review で未公開、GameReviewQueueで人間審査
- **テスト354件** + contract.consistency.test.ts（contract.tsが唯一の正であることの強制）
