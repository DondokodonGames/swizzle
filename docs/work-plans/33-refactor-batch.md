# WP33: 承認済みリファクタの実行

**担当**: Opus / **依存**: WP32の承認シート（人間チェック済み）+ WP31のCI / **想定規模**: L（項目ごとにセッション分割を推奨）

---

以下をそのまま新しいOpusセッションに貼って実行する。冒頭で `docs/architecture-review.md` の**承認シートでチェックされた項目のみ**を確認し、未承認項目には触れないこと。

## 背景

WP32のアーキテクチャレビューで提案され、人間が承認した項目を実行する。
CI（tsc+test+lint、WP31）がgreenであることが各ステップの守り。

## 進め方（厳守）

- **1承認項目 = 1コミット**（大きい項目は複数コミット可だが、項目をまたぐコミット禁止）
- 各項目の完了条件: `npx tsc --noEmit --skipLibCheck` クリーン + `npm run test` 全件green + `npm run lint` exit 0
- **挙動を変えないリファクタに徹する**。挙動変更が必要だと判明したら、その項目を中断して報告（勝手に仕様を変えない）
- 大物（EditorGameBridge分割・AI v2分割）は、着手前に対象範囲のキャラクタライゼーションテスト（現挙動の固定テスト）を先に書く
- `src/types/editor/contract.ts` と `src/services/rule-engine/` の変更が必要な項目は、変更内容を要約して人間の確認を取ってから実施

## 事前定義済みの候補（WP32の承認シートに含まれる想定。承認されたもののみ実行）

1. **ReviewQueue/GameReviewQueueの共通化**（S-M）: プレイ→評価→次へ のフロー・フィードバックパネル・ブリッジ操作を共有フック/コンポーネントに抽出。データソース（ローカルJSON vs Supabase）だけを差し替え可能に
2. **GameLoadingService 新設**（M）: 4経路の読込を統一（WP32の設計に従う）。PublicGame→GameProject変換を1箇所に
3. **重複型の統合**（S）: GameScore等、複数ファイルに重複定義された型を共有モジュールへ
4. **flag条件不整合の恒久対応**（M）: WP32で選定された方式を実装（テスト必須: v2形のflag条件がエンジンで正しく評価されること）
5. **死蔵コードの削除**（S）: WP31のTODO棚卸しで「削除」判定されたもの+src/marketing/のstub群（人間が削除承認した場合のみ）

## やらないこと

- 未承認項目・新規の思いつきリファクタ
- 機能追加（純粋なリファクタのみ）
- WP22（UI刷新）と同時進行（同じファイルに触れるため。22完了後に実施）

## 受け入れ基準

- 承認済み項目がすべて完了 or 「中断+理由」が報告されている
- 各コミットでCI green
- 削除されたコードの参照が残っていない（ビルド成功）

## 検証方法

1. 各項目後にCI 3点セット
2. リファクタ対象の画面/フローを `npm run dev` で実際に操作（エディタ起動・ゲームプレイ・レビューUI）
3. 完了時: 項目ごとの成果サマリ+変更ファイル一覧を出力
