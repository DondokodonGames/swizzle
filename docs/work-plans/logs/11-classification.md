# WP11: プロンプトダイエット — 制約の3分類

`SpecificationGenerator.SPEC_PROMPT`（2,300行超）と `GameConceptGenerator.CONCEPT_PROMPT`
の全セクションを、次の3分類で棚卸ししたもの。lean 版プロンプト設計の根拠。

- **ハード制約（残す・簡潔に）**: エンジン上不可能 / 自動検証で必ず落ちる / 人間の操作限界。
- **ソフト指針（短い例示に圧縮）**: 推奨であって強制ではないもの。lean では強制表現（絶対/必ず/禁止）をやめる。
- **削除候補（モデルを信頼して削る）**: 使いすぎ禁止リスト、長大なテーマ例、冗長な重複警告。

守りはプロンプト外に既に存在する（類似度ゲート70%、PhaseCompiler+グラフ検証、DryRunSimulator、
LogicValidator/LogicRepairer、公開ゲート、テスト354件）。「壊れたゲームは検証層が落とす」ため、
プロンプトは発想を広げる方向へ振り直せる。

## やらないこと（守りは触らない）

- classic 版プロンプトは削除せず、`PROMPT_STYLE` で両立（A/B 比較完了まで）。
- 類似度ゲート・フェーズ層・バリデータ・公開ゲートは変更しない。
- `mechanics-catalog.json` の内容は変更しない（lean では選択を「強制」から「提示」に変えるだけ）。
- `contract.ts` / rule-engine は変更しない。
