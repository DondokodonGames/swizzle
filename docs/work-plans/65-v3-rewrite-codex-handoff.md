## 2. 修正方針(設計 — 全ゲーム共通・最重要)

機械ゲート(validator/scorer/smoke)は**通っても品質を保証しない**(本セッションで、ゲートを全通過した13本が「ほぼ0点」と評価された)。
以下の5原則を**ゲーム全体で**満たすことが本丸:

### 2.1 世界とメカニクスの一致(最重要)
その世界で「**自然にやりたくなる行動**」が勝ち筋と一致すること。
- ✗ 反例: 「爆弾解除」なのに「カウンタが1になるまで待ってタップ」= 爆弾は早く解除したいので矛盾。
- ✓ 例: 「駅に電車を停める」= ちょうどの位置で停めるのが自然な目的(早すぎ=手前、遅すぎ=行き過ぎ、両方まずいのが直感的)。

### 2.2 画でルールと因果を伝える
何をする / 何が成功 / 何が失敗を、**抽象的な数字やコード上の規則ではなく、背景・設置物・状態変化**(あふれる・ぶつかる・通り過ぎる・落ちる)で一瞬で分からせる。
- **失敗の"まずさ"は必ず画で起こす**(コード内部の判定でなく、見て分かる物理的結果)。
- ✗ 「数字が0になると失敗」= 0の不味さが画になっていない。
- ✓ 「電車がホームを通り過ぎて客が乗れない/トンネルで止まって隙間」= 失敗が画で起きる。

### 2.3 操作の可視化
何をタップ→何が起きるかを、**見た目のアフォーダンス + ATTRACT実演**で分からせる。
- ✓ 例: 画面下の**ブレーキレバー**(タップ対象が見える)→ 引くと電車が減速して止まる、の因果を実演で提示。

### 2.4 遊びの中の判断
**ただ待つを禁止**。変化するものを読んで**先読み・判断**する余地を、**直感的に**与える。
- ✗ 「その瞬間にタップ」だけ = 待つだけで理不尽。
- ✗ 抽象UI(予測マーカー等)の読解 = ユーザーに伝わらない。
- ✓ 速度が毎回変わる電車を見て、早めにブレーキを判断(先読み)。

### 2.5 テーマ = 見れば何をすべきか分かるもの
テーマは舞台名ではなく、**テーマを見れば行動が分かる**もの。**文字での説明(操作説明・ルール文)は厳禁**(タイトル/§3ホワイトリスト以外のテキスト禁止)。

### 参考: 017「ぴたっと駅停め」(timing_one_shot の良設計例)
- **世界とメカ一致**: 客の待つ駅ホームに電車をちょうど停める。
- **画で因果**: 停止ゾーン(緑)/待つ客/行き過ぎ(通過)・手前(隙間)が全部画で起きる。
- **操作可視化**: 下部のブレーキレバーをタップ→電車が減速。ATTRACTで手がレバーを引く実演。
- **判断**: 進入速度が毎回変化(加速型)→ 早めのブレーキを先読み。
- ※ この設計のアートは仮。Codex がドット絵化して完成させる。

## 3. 修正方針(アート — Codex担当の品質基準)

- **目標: MOTHER3(GBA)級のドット絵**。実際にドット絵である必要はないが、**「ドット絵らしく・面白そうに」見えること**を優先(ドット絵らしさは"ゲームである"ことの記号)。
- **判断はゲーム全体(合成された動く画面)で**。単体スプライトの巧拙では判断しない/できない。
- **主役(操作対象)は大きく前面に**。焦点の対象が小さいと「対象として成立しない」(017で電車を小さくして悪化した実測あり)。
- **記号感(一目で何か分かる)と高品質を両立**。装飾物の**量**を増やすのではなく、**必要要素の質**を上げる。
- **成功/失敗は思いっきり差をつける**(表情・演出・結果が画で激変。祝祭 vs 崩壊)。
- **手続き描画で無理に作り込まない**。アートは**実アセット(ドット絵画像)を `game.draw.image` で描く**。手続き描画は FX / UI / telegraph / 画面演出に限定。
- **アート方針はゲーム内容ごとに最適化**(スタイルパックは出発点であって縛りではない)。

## 4. 受け渡しの土台(Claude 側で別途整備)

Codex 生成のドット絵がそのまま載る配線。**本書の方針確定後に実装**する:
1. **エンジン**: `game.draw.image` に**スプライトシートのコマ切り出し(source rect)+反転**を追加 → 歩行/表情アニメ可能に。
2. **ランナー/smoke**: ゲームの**宣言アセット(マニフェスト)を読み込む**(現状 smoke は `assets:[]`)。
3. **アートブリーフ(Claude出力)**: ゲームごとに必要スプライト(id/役割/pxサイズ/コマ/アンカー/透過/スタイル方針)を構造化して出力 → Codex が満たす。
4. **契約チェック**: validator が「`draw.image` の参照 id が全てマニフェストに存在」を検査、smoke は欠損/空描画で FAIL。

## 7. このセッションの実測知見(Codexへの申し送り)

書き換えで**やってはいけないこと**(本セッションで実際に外した点):
1. 手続き描画の「陰影付き矩形」や低解像度スプライトは**品質不足**(のっぺりして"造形品質が上がらない")。
2. **装飾物(月・星・時計・ランプ等)を増やしても品質は上がらない**。必要要素の質を上げること。
3. **抽象UI(予測マーカー等)は「判断」として伝わらない**。直感的な読みにする。
4. **主役が小さい**と操作/対象として成立しない。大きく前面に。
5. **単体アートでは品質判断不可**。必ずゲーム全体(動く合成画面)で作り込み・評価する。
6. **文字でゲーム名やルールを説明しない**。背景・設置物・実演で伝える。
